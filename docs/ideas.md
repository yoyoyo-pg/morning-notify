## 開発計画タスク

### Notion 日次ページ記録

朝の通知実行時に、天気・予定・ニュースを Notion へ自動記録する。

**やりたいこと**
- 毎朝の通知と同時に Notion データベースへページを作成
- 天気・Google カレンダーの予定・ニュースヘッドラインを記録
- 後から振り返れる朝の記録として蓄積する

**Notionページのイメージ**
```
2026/04/27 朝の記録

### 🌤 天気
・晴れ　22°C（最高28°C / 最低18°C）　降水確率10%

### 📅 今日の予定
・10:00　チームミーティング
・14:00　1on1

### 📰 ニュース
・【政治】〇〇〇〇
・【経済】〇〇〇〇
```

**必要な準備**
- `NOTION_API_KEY` — Notion インテグレーションページで発行
- `NOTION_DATABASE_ID` — 記録先データベースのページ ID
- `notion-client>=2.0` を requirements.txt に追加

**実装メモ**
- `src/notion_logger.py` を新規作成
- `main.py` でデータ取得を上位に集約し Discord 通知・Notion 記録で共有
- Secrets 未設定時は except でスキップし、Discord 通知は正常動作を維持する

### イベント・地域情報通知

名古屋のイベント情報を定期的にDiscordへ通知する。毎日の朝通知とは別チャンネルで、週2回程度の配信を想定。

**やりたいこと**
- 久屋大通のイベント情報
- 名古屋市内のイベント全般（週末のイベント、フェス、マルシェ等）

**実装イメージ**
```
# 週2回、#events チャンネルへ
🎪 今週末の名古屋イベント情報
　・〇〇マルシェ（久屋大通公園 / 5/3〜5/4）
　・〇〇フェスティバル（鶴舞公園 / 5/4）
```

**技術的な検討事項**

情報ソース候補：
| ソース | 特徴 | 取得方法 |
|--------|------|---------|
| 名古屋市観光情報サイト | 公式情報 | スクレイピング or RSS |
| 久屋大通公園公式サイト | ピンポイントで欲しい情報 | スクレイピング |
| Peatix | 有料・無料イベントが豊富 | RSS あり |

スケジュール案：月・木の朝6時（`0 21 * * 1,4` UTC）

**Discord チャンネル構成（案）**
| チャンネル | 内容 | 頻度 |
|-----------|------|------|
| `#morning` | 天気・カレンダー・ニュース | 毎日 |
| `#events` | 名古屋イベント情報 | 週2回 |

**懸念点**
- スクレイピングはサイト構造の変更で壊れやすい
- 公式 RSS があるかどうかが鍵
- イベント情報の鮮度管理（過去のイベントを通知しないようにする）

## AI 運用・ワークフロー改善メモ

このプロジェクトを通じて得た AI との協働ノウハウ。今後の開発運用に活かす。

### ワークスペースを用途ごとに分ける

1つの巨大なプロジェクトに全部入れるのではなく、用途ごとに小さく分ける。

| ワークスペース | 用途 |
|-------------|------|
| スキル・自動化用 | スクリプト、ワークフロー |
| 知識管理用 | メモ、調査結果 |
| 資産管理用 | 財務・所有物の記録 |
| 日々のタスク管理用 | TODO、進捗 |

共通点：Markdown で情報を残し、Claude Code から読ませたり更新したりする。

### 複雑な作業は役割分担させる

複雑な作業を1つの長いプロンプトで全部やらせるのではなく、役割を分けて進める。

```
指揮役
 |
実施管理役
├─ 調査役
├─ 実装役
├─ レビュー役
└─ 引き継ぎ整理役
```

**効果**
- 1つの会話に情報を詰め込みすぎずに済む
- 調査だけ・レビューだけを任せる使い分けがしやすい

**注意点**
- 単純な修正なら1セッションで進めた方が早い
- 分担が効くのは、調査範囲が広い作業や、レビュー観点を分けたい作業

### 作業ドキュメントの構成

作業の流れを以下のMarkdownファイルで残す運用。

| ファイル | 内容 |
|---------|------|
| `plan-XXX.md` | 雑な方針・メモ |
| `spec-XXX.md` | 構造化した仕様 |
| `todo.md` | 作業タスク |
| `lessons.md` | ハマったこと・次回への教訓 |

**重要な考え方：「次回の自分は、ほぼ他人」**

AIとの作業はその場では進むが、後から見ると「なぜこうしたのか」が分からなくなりがち。
設計・判断理由・ハマりどころ・次にやることをMarkdownに残す運用にする。
